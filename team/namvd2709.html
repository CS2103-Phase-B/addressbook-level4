Adding an appointment: appoint

Since v1.3

Add an appointment to a person in the addressbook.
Format: appoint INDEX ap/date (dd/mm/yyyy) time (hh:mm) duration(minutes)
Example: appoint 1 ap/12/12/2020 15:12 60

  • If the person already have an appointment, the new appointment will override the old one

  • If ap/ is left blank, the old appointment will be deleted.

End of Extract

Justification

As a student, the target user of this application will have a lot of appointments for group work, dating, setting up
meetings and appointments with other students and professors. Hence the need to store appointment details.

Implementation

Start of Extract [from: Developer Guide]

Appointment

Appointment is stored in a UniqueAppointmentList, which makes sure that the appointment is unique and does not clash with other
appointments.

Design Considerations

Aspect: Storing appointments
Alternative 1 (current choice): Storing appointment using an in-memory UniqueAppointmentList.
Pro: Does not take up space.
Cons: Might be slow if there is a huge number of appointments (which is unlikely for a student)
Alternative 2: Storing appointment using a separate XML file
Pro: Will be faster if theres many appointments
Cons: Harder to understand. Does not follow convention of other commands.

Aspect: Number of appointments
Alternative 1 (current choice): Allows 1 appointment per contact.
Pro: Easier to manage (add, remove, overwrite, etc.)
Cons: Is limited in case user have multiple appointments with others.
Alternative 2: Allows multiple appointment per contact.
Pro: Catered to users with multiple appointments per contact.
Cons: Would be hard to view on address book person card, hard to manage.

End of Extract

Locating persons by name: find (alias f)

Finds persons whose names starts with the given keywords.
Format: find KEYWORD [MORE_KEYWORDS] or f KEYWORD [MORE_KEYWORDS]

  • The search is case insensitive. e.g hans will match Hans

  • The order of the keywords does not matter. e.g. Hans Bo will match Bo Hans

  • Only the name is searched.

  • Name will be matched from the start of the name. e.g Han will match to Hans

  • Persons matching at least one keyword will be returned (i.e. OR search). e.g. Hans Bo will return Hans Gruber, Bo Yang

Examples:

  • find John
    Returns john and John Doe

  • find Betsy Tim John
    Returns any person having names starting with Betsy, Tim, or John

  • f Na
    Returns Nam (if Nam is the only close match).

Implementation

Start of Extract [from: Developer Guide]

Searching mechanism

Here, searching entails the autocomplete function, finding mechanism and suggesting mechanism. The search mechanism is done with user’s usability in mind and is fully implemented such that it is most intuitive to use for the users.

Autocomplete function

The autocomplete function mimics the behaviour of familiar Unix-based command lines. The autocomplete function is managed by a class AutocompleteManager, whose function attemptAutocomplete searches for the string starting with the specified characters in the string array provided and will return the string if a unique match is present.

Finding mechanism

The find function currently matches name in the address book if those name starts with the find keyword. Whenever the a find event is triggered, the address book will filter the address book based on the predicate provided, the mechanism of which is specified by NameContainsKeywordsPredicate class. This class overrides the test function to filter the list based on containsWordsStartWithString, a StringUtil method which returns whether a string starts with another string. For example, 'Dav' will return 'Dave'
Looking forward, the finding mechanism will be more flexible, by implementing fuzzy search mechanism which allows returning close matches. For example, 'Dve' will return 'Dave' or 'Dove'.

Suggesting mechanism

The suggestion mechanism will be a hybrid of the above two functionalities, in which suggestion, whether commands or a match string inside the addressbook will be shown as the user type commands. This has not been implemented but will be a UI-heavy component that anchor on the fuzzy search logic.

End of Extract
