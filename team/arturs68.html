Grouping persons : group

Since v1.2

Used to add to a group with GROUP_NAME a person identified by the index used in the last person listing.
If such a group does not exit, it is created.
Format: group INDEX gn/GROUP_NAME

  • If a group with such a name exists, adds the persons at INDEX.

  • The index refers to the index number shown in the most recent listing.

  • The index must be a positive integer 1, 2, 3…​

Examples:

  • list
    group 2 gn/CS2103
    Adds the 2nd person to the group CS2103 in the address book.

  • find Betsy
    group 1 gn/Family
    Adds the 1st person from the results of the find command to the group Family.

Removing from groups : ungroup

Coming in v2.0: ungroup

Used to remove from a group person identified by the index used in the last person listing.
Format: ungroup INDEX gn/GROUP_NAME

  • If a group ends up being empty, it is removed.

  • The index refers to the index number shown in the most recent listing.

  • The index must be a positive integer 1, 2, 3…​

Examples:

  • list
    ungroup gn/CS2103 2 3
    Removes the 2nd and 3rd person from the group CS2103 in the address book.

  • find Betsy
    ungroup gn/Family 1
    Removes the 1st person from the results of the find command from the group Family.

End of Extract

Justification

The user has to manage many contacts most of which can be classified for efficient retrieval and processing.

Design Considerations

Aspect: Storing information about group members
Alternative 1 (current choice): Members store list of groups they belong to
Pros: Code related to the tags (filtering, storing) can be reused
Cons: Finding the members of specific group more computationally expensive.
Information about groups has to be added wherever Person is used (add, edit commands). Some code duplicated
Alternative 2: Group stores list of members
Pros: Easier to understand. Easier to retrieve all the members of a group
Cons: Even though some mechanisms are similar to ones related to Tag, new classes are needed to handle them.

Aspect: Managing groups
Alternative 1 (current choice): Groups can be managed (members added, removed) only using group-specific commands
Pros: Avoids cluttering other commands (add, edit). Separates the concepts of Person and Group, also Tag and Group.
Cons: Adding a Person to a group at the moment of the Person’s creation is impossible
Alternative 2: Groups are accessed and managed just as Tags, when managing a Person
Pros: Avoids creation of a new command.
Cons: Adds another attribute to a Person increasing its complexity. Not intuitive to manage group by managing Person.

Start of Extract [from: User Guide]

Changing profile picture : changepic (alias pic)

TBI in v2.0: storing the pictures by the app

Changes the profile picture of the person specified by the index
Format: changepic INDEX p/PICTURE_PATH

The picture has to be present on the drive of the computer in order to be displayed. If the picture cannot be found,
it is not shown. To choose default picture, type default_pic.png as a path.

Examples:

  • pic 2 p/C:\Users\User\Documents\mypic.jpg

  • changepic 1 p/default_pic.png

End of Extract

Justification

The user wants to store the pictures together with the contacts, so that she recalls the people easier

Implementation

Start of Extract [from: Developer Guide]

Profile Pictures

Profile pictures are meant to be stored by app (in v.1.5 by the user at his own discretion) and they are added to the app by specifying the correct path to the file. In case the picture cannot be accessed later, it is not displayed.

Design Considerations

Aspect: Place of picture storage
Alternative 1 (current choice): Pictures are stored by the app after a successful load
Pros: In case the user moves the pictures or the app to another device, pictures are not lost
Cons: Data may be duplicated on the user’s computer (stored both by the app and the user). Implementation harder.
Alternative 2: Pictures stored by the user
Pros: Easier to implement. Easier to understand
Cons: May cause unwanted behavior if user is unaware and moves the pictures from the original location

Aspect: Format of stored pictures
Alternative 1 (current choice): Pictures are stored in their original format
Pros: Easier to understand. User can access and modify the stored data easily
Cons: Differs from storage of the rest of the data
Alternative 2: Pictures are stored in the xml files, just like all the other data
Pros: Follows convention
Cons: Not possible for the user to access the stored data directly not using the app. Encoding and decoding needed

End of Extract
